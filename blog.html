learnyounode, Prototypical Inheritance, modules

Tuesday, July 28 2015

While I am new to JS, it is quite apparent that prototypical inheritance is a tricky subject. I have been reading Kyle Simpson's You Don't Know Javascript and watching his talks on pluralsight which have really made things clear for me. He talks about this subject in terms of behavioral delegation rather than traditional class-inheritance, and object-linkages rather than OOP. I won't go into too much detail about that here, but I highly recommend it.

While working on learnyounode modules I wanted to understand the linkage of modules that I directly and indirectly working with.

For example, in the httpclient exercise we work directly with the http module in order to (1) make a url request, (2) receive a response object and then (3) collect data from this response object. The code to this is fairly short and sweet:

var http = require('http');

http.get(url, function (response) {
response.setEncoding('utf8')
response.on('data', function(data) { console.log(data); });
response.on('error', console.error)
})

So we use the http module to make a get request and the callback function returns a response object. That response object will emit data events which we will capture and print to console.
We use a setEncoding method so that we get back actual strings instead of a buffer objects.

To fully understand whats going on here a few things should be explained.

  1. Some objects in node emit events. (examples: fs.readStream emits an event when the file is opened, net.Server emits an event each time a peer connects to it)

  2. All objects that emit events are instances of events.EventEmitter.

  3. A stream is an "interface" in node --- If you were to ask Kyle Simpson, I think he would debate this is incorrect terminlogy and better to say a stream object has linkages from many objects

  4. Stream objects emit events && thus links to events.EventEmitter

  5. An http response to a server is a stream object (it "implements"/links to the stream object

So.. my question: What does the prototype inheritance diagram look like here?

If I use the .__proto__ method on some of my objects to try and figure this out but it left me even more confused!
For example, if I go up 1 link on the prototype chain..
console.log(response.__proto__);
{ setTimeout: [Function],
read: [Function],
_read: [Function],
destroy: [Function],
_addHeaderLines: [Function],
_addHeaderLine: [Function],
_dump: [Function] }
If I go 2 links up the prototype chain.. I get somewhere that makes sense to me: the Stream object**.

console.log(response.__proto__.__proto__);
{ push: [Function],
unshift: [Function],
isPaused: [Function],
setEncoding: [Function],
read: [Function],
_read: [Function],
pipe: [Function],
unpipe: [Function],
on: [Function],
addListener: [Function],
resume: [Function],
pause: [Function],
wrap: [Function] }

If I go up 3 links. I am expecting to see the events.EventEmitter object (see #4). But instead, more confusion:

console.log(response.__proto__.__proto__.__proto__);
{ pipe: [Function] }

**The only way I know this is a stream object is because I read the API and the steam object has all these methods. Is there not a better way??

Women Who Code NYC Meetup -- Lightning Talks

Tuesday, July 7 2015

Bienvenidos a Nueva York! I intend to keep most of my posts technical, but for my first one I want to write about the Women Who Code meetup I had the opportunity to attend yesterday.

I saw 10 lovely ladies speak. It was my first event like this with all women, and I loved it. Of course diversity of all kinds of is necessary for great collaboration, but there is something very relaxing about working with women, as a woman. One of the reasons I love programming so much is because of the lack of discrimination and BS that, comparitively, seems to accompany other professions. In programming I often feel the premise tends to be, "Can you make it work? Great." No one cares who you are, what you look like, where or even if you went to college, just prove you have the skills and enjoy making things work. Being surrounded by action-oriented women extends the attitude of minimizing BS. It inspires creation and acceptance --even encourgement-- of failure to dominate the culture of thinking. (Maurya Couvares, the founder of ScriptEd , even spoke about the importance of learning how to fail in the context of her company). Its just nice to be with your peeps.

Speakers discussed the technical-- js promises, ES6 features , overview of Swift, web accessibility -- and nontechnical alike.

One thing worth noting were 2 vibrant ladies who have recently started the company monarq-- a friendship app for women. Even more interestingly, they also started a women's hackathon which it think is fan-freaking-tastic. I definitely intend on going to in October.

Another women's oriented world-wide collaboritive event that was discussed was Womens Entrepreneurship Day .
